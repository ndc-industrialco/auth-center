# ปัญหาที่พบบ่อย

ใช้หน้านี้เมื่อ:

- grant role แล้ว แต่แอปยังขึ้น `Access Denied`
- role ใน Auth Center ถูกแล้ว แต่ consumer app ยังมองไม่เห็น
- user login สำเร็จ แต่ถูกพาไปผิดหน้า
- token claim ไม่ตรงกับที่คาดไว้

## 1. Role ถูก grant แล้ว แต่แอปยังเข้าไม่ได้

อาการ:

- Auth Center admin เห็น role ถูกต้อง
- consumer app ยัง deny access

สาเหตุที่พบบ่อยที่สุด:

- token ที่ browser ใช้อยู่ยังเป็น token เก่า

วิธีเช็ค:

1. ตรวจว่าผู้ใช้ถูก grant role ให้ app ที่ถูกต้อง (`appId` ตรง)
2. decode token ปัจจุบันของ consumer app
3. ดูว่า `appRoles` มี role ใหม่จริงหรือไม่

ถ้า token ยังไม่มี role ใหม่:

- ให้ user sign out และ sign in ใหม่
- หรือใช้ refresh session ถ้าแอปรองรับ

## 2. Role ใน Auth Center ถูก แต่ token ไม่มี role

อาการ:

- ใน admin UI grant ถูกแล้ว
- ใน token `appRoles` ว่าง หรือเป็น role เก่า

สาเหตุ:

- role change ไม่ได้แก้ token เก่าที่ยิงไปแล้ว

สิ่งที่ต้องจำ:

- Auth Center เก็บ role
- consumer app authorize จาก JWT claim
- token เก่าไม่ update เองอัตโนมัติ

## 3. Audience ไม่ตรง

อาการ:

- verify token ไม่ผ่าน
- หรือ verify ผ่านบาง flow แต่ role ไม่ขึ้นตามที่คาด

วิธีเช็ค:

- `AUTH_CENTER_AUDIENCE` ใน consumer app ต้องตรงกับ `appId` ที่ register ใน Auth Center
- role grant ต้องถูก grant ให้ app เดียวกันนั้น

ตัวอย่าง:

- app register เป็น `qms`
- `AUTH_CENTER_AUDIENCE` ก็ต้องเป็น `qms`

## 4. Grant role ให้ผิด app

อาการ:

- ผู้ใช้มี role จริง แต่เป็นของอีก app

วิธีเช็ค:

- เปิด Auth Center admin
- ดูว่า role grant ผูกกับ `appId` ไหน
- ยืนยันว่าเป็น app เดียวกับ consumer app ที่กำลังทดสอบ

## 5. Route Guard เช็คชื่อ role ไม่ตรง

อาการ:

- frontend แสดงว่ามี role
- แต่ middleware หรือ API ยัง deny

วิธีเช็ค:

- middleware เช็ค role ชื่อเดียวกันหรือไม่
- backend API เช็ค role ชื่อเดียวกันหรือไม่
- ชื่อ role case-sensitive ตรงกันหรือไม่

## 6. Login แล้วโดนพาไปผิดหน้า

อาการ:

- login สำเร็จ แต่กลับไป `Auth Center` root
- หรือกลับไป `localhost` ทั้งที่ควรกลับ consumer app

สาเหตุที่พบบ่อย:

- consumer app ส่ง user ไป endpoint ผิดของ Auth Center
- `redirectUri` ตั้งผิด
- consumer callback ไม่ได้เป็นตัวสร้าง session แล้ว redirect ต่อ

flow ที่ถูกต้อง:

1. consumer app redirect ไป `Auth Center` login page
2. Auth Center login สำเร็จ
3. Auth Center redirect กลับ `consumer callback`
4. consumer callback verify token
5. consumer callback สร้าง session cookie
6. consumer callback redirect กลับหน้าปลายทางของแอป

## 7. LOCAL session ไปเรียก Microsoft Graph ในนามผู้ใช้

อาการ:

- app พยายามใช้ delegated Graph ทั้งที่ user login ด้วย employeeId + password

กติกา:

- LOCAL session ต้องไม่ใช้ delegated Graph
- เช็ค:
  - `authMethod === 'ENTRA'`
  - `canSendDelegatedMail === true`

ก่อนเรียก delegated Microsoft Graph

## 8. Checklist เร็ว ๆ เวลา debug

- [ ] User ถูก grant role แล้วจริง
- [ ] Grant อยู่บน `appId` ที่ถูกต้อง
- [ ] `AUTH_CENTER_AUDIENCE` ตรงกับ `appId`
- [ ] Token ปัจจุบันมี `appRoles` ที่ถูกต้อง
- [ ] User sign in ใหม่แล้วหลัง role เปลี่ยน
- [ ] Middleware เช็ค role ถูกชื่อ
- [ ] Backend API เช็ค role ถูกชื่อ
- [ ] `redirectUri` ชี้กลับ consumer callback ถูกต้อง

## 9. สรุป

ปัญหาที่พบบ่อยที่สุดมี 3 กลุ่ม:

1. token เก่า
2. grant ผิด app
3. `redirectUri` หรือ callback flow ตั้งผิด

ถ้า role ถูกใน Auth Center แต่แอปยัง deny access ให้เริ่มจากการตรวจ token ปัจจุบันก่อนเสมอ
